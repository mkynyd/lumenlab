# MiniMax M 系列模型使用技巧

> 来源: https://platform.minimaxi.com/docs/token-plan/prompting-best-practices

Token Plan 模型使用技巧，涵盖 prompt 模板、工具调用、Agent 工作流和长上下文。Token Plan 模型为纯语言模型，图片、音频、视频需先用其他工具处理后再传入 prompt。

---

## 通用原则

### 1. 指令明确具体

模型在目标、约束和输出要求明确时表现最好。说明要做什么、优先考虑什么、什么样的结果是好的。

**黄金法则**：把提示词分享给不熟悉任务的人，如果他们感到困惑，模型也会。

### 2. 补充约束的意图

解释**为什么**有某个约束，帮助模型做出更好的权衡。适用于格式化、安全、无障碍、工作流约束等场景。

> "当模型可能只满足字面要求、忽略真实使用场景时，请补充说明意图。"

### 3. 用示例引导输出

准备好的 few-shot/multishot 示例通常比抽象的风格描述更稳定：
- **相关**：贴近真实用例
- **多样**：覆盖边界情况和模糊输入
- **具体**：展示实际输出风格，不仅是主题

> "复杂模式匹配任务请给 3-5 个差异化示例。重复同类示例只会浪费 token，无法教会模型新东西。"

### 4. 使用 Prompt 模板

对重复任务，将提示词写成具名变量的可复用模板。方便多输入测试、版本对比和长期稳定行为维护。

### 5. 匹配输出语言

多语言输入或需要指定输出语言时，明确说明。例如："即使素材是英文，也请用中文回复。"

## 输出与格式

### 用清晰的段落结构

使用标签分段（如 `任务`、`背景`、`素材`、`约束`、`输出格式`），保持结构扁平，避免深层嵌套。

### 设定角色、格式与长度

- 角色定义同时描述专业背景、工作范围和判断标准
- 使用显式输出契约：段名、表列名、条目数量限制、范围边界
- 避免模糊表述如"详细一点"、"简短一点"

## 长上下文

### 任务放在素材之后

**所有长上下文技巧中最重要的一条**：任务越靠近模型回答位置，越容易保持关注。

### 为素材建索引并分隔

为每个文档标记名称+日期，要求模型在回答前先引用或总结各文档的关键内容。来源标注减少噪音，便于验证。

## 工具调用

### 工具定义

为每个工具定义名称、用途、输入、返回结构、失败处理。模型应在了解工具契约后再决定是否调用。

### 并行工具调用

独立工具调用（如多来源搜索）可以并行；有依赖的调用（如先查询再更新）应顺序执行。

> "独立的只读查询适合并行。像'先找到客户，再更新该客户记录'这样的工作流应保持顺序执行。"

### 避免过度调用

指定停止规则：工具应在"实质提升回答质量"时使用，而非为了显得忙碌。需要确认才能执行破坏性操作（删除、购买、发送等）。

## 思考与推理

### 控制推理深度

- 规划、调试、权衡、长链执行 → 请求深入分析
- 提取、重写、格式化 → 指示模型直接回答

简单请求示例：`从下面文本中抽取公司名称。只返回 JSON 数组，不需要解释。`

### 降低幻觉

三种方法：
- **允许拒绝** — 告知模型无法确认时的回应方式
- **来源引用** — 要求模型标注使用材料的来源
- **先约束、再生成** — 在任务前指定允许的来源/时间范围/产品版本

## 长任务与 Agentic 工作流

- 每次只给模型少量当前目标
- 在单窗口内维护状态（计划、进度、待解决问题）
- 分阶段处理时，第一窗口构建框架，后续窗口迭代
- 使用测试文件跟踪长时间迭代
- **警告**：使用压缩上下文的工具时（如 Claude Code），保持 system prompt 简洁，模型在临近上下文容量阈值时可能提前终止任务

推荐长任务 system prompt：
```
这是一项非常冗长的任务。请充分利用可用上下文窗口。
进入下一部分前，请先彻底完成当前部分，避免任务完成前耗尽 tokens。
```

## 评估与迭代

管理重要 prompt 应如同管理产品配置：
1. 定义成功标准（正确性、格式、工具调用准确性、语气）
2. 准备 10-30 个代表性测试案例（含边界情况）
3. 用新旧 prompt 运行同一批测试
4. 对比输出，记录退步
5. 仅当候选版本改进了目标指标且不破坏必需行为时才更新
6. 记录简短变更日志（变更内容、原因、改进的示例）
