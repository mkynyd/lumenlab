# DeepSeek 上下文硬盘缓存 (KV Cache)

## 概述

上下文硬盘缓存是 DeepSeek 的一项自动优化技术，能在多轮对话和长文本处理中显著降低延迟和成本。该功能**对所有用户默认开启**，无需任何代码修改。

## 缓存机制

缓存以**缓存前缀单元**为基本单位落盘。后续请求必须**完整匹配**某个已落盘的缓存前缀单元才能命中。

### 落盘触发方式

1. **请求结束位置落盘**: 每次请求的用户输入结束处和模型输出结束处各产生一个缓存前缀单元
2. **公共前缀检测落盘**: 系统发现多次请求的公共前缀后，将其作为独立单元落盘
3. **固定 token 间隔落盘**: 在长输入/输出中按一定 token 间隔截取单元

## 命中示例

### 多轮对话

- 第 1 轮: `A + B` → 缓存 `A` 和 `A + B`
- 第 2 轮: `A + B + C` → 命中 `A + B` 单元 ✅
- 第 2 轮备选: `A + C` → 不命中 ❌（公共前缀 `A` 将被单独落盘）
- 第 3 轮: `A + D` → 命中 `A` 单元 ✅

### 长篇问答

- 前两次请求（相同 system prompt + 相同长文档但不同问题）：不命中
- 系统识别公共前缀（system + 文档）并落盘后，第三次请求即可命中

## 计费

缓存命中与未命中有不同的 token 定价，详见 [模型 & 价格](./deepseek-models-pricing.md) 页面。

## 查看命中情况

API 返回的 `usage` 字段包含两项新数据：

```json
{
  "usage": {
    "prompt_tokens": 1000,
    "completion_tokens": 200,
    "total_tokens": 1200,
    "prompt_cache_hit_tokens": 800,
    "prompt_cache_miss_tokens": 200
  }
}
```

- `prompt_cache_hit_tokens`: 缓存命中的输入 tokens 数
- `prompt_cache_miss_tokens`: 缓存未命中的输入 tokens 数

## 注意事项

- 只匹配用户输入的**前缀部分**
- 输出仍受 `temperature` 等参数影响，效果与不使用缓存一致
- 缓存系统"尽力而为"，不保证 100% 命中率
- 缓存构建耗时为秒级
- 闲置缓存会在几小时到几天内自动清空
