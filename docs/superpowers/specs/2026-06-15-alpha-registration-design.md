# Alpha Registration and Credential Distribution Design

## Summary

Alpha access is controlled by registration codes created in a separate
`course-ai-regadmin` application. The admin application and `course-ai-lab`
run as separate containers, use separate PostgreSQL databases, and expose
different subdomains:

- `lab.mkynstudio.top`
- `regadmin.mkynstudio.top`

The admin application is the source of truth for credential profiles,
registration codes, audit events, and publication history. The main
application is the source of truth for users, redemption counts, access
revocation, and runtime provider access.

## Confirmed Product Decisions

- A registration code has a configurable maximum redemption count.
- Expiration is optional.
- Codes are securely generated by default; strong custom codes are allowed.
- Credential profiles are reusable and contain a required DeepSeek key and an
  optional MiniMax key.
- Provider keys must pass a minimal provider request before publication.
- Changes remain drafts until the administrator explicitly publishes them.
- Stopping code redemption and revoking existing users are separate actions.
- Existing test users and personal API keys are backed up and removed before
  Alpha; every Alpha user registers again.
- Email verification and application-level usage quotas are out of scope for
  the first Alpha.

## Security Architecture

The admin database encrypts provider keys and recoverable registration codes
with its own AES-256-GCM key. The main database never stores registration code
plaintext. It stores an HMAC-SHA-256 digest produced with a main-app-only
pepper.

Publication uses a versioned push API over the private Docker network:

1. Regadmin decrypts publishable secrets only in server memory.
2. It encrypts the JSON snapshot with a random AES-256-GCM data key.
3. It wraps that data key with the main application's RSA-OAEP public key.
4. It signs the exact request body with HMAC-SHA-256 using a separate sync
   authentication secret and sends timestamp and nonce headers.
5. The main application verifies timestamp, signature, nonce, and publication
   id before decrypting.
6. The main application re-encrypts provider keys with its own AES key and
   applies the snapshot in one database transaction.

The public registration endpoint accepts email, password, and registration
code. It normalizes and hashes the code, atomically consumes one redemption,
creates the user, records the redemption, and binds the user to a credential
profile. Provider keys are resolved only in server-only data access code and
are never returned to a browser.

## Administration

Regadmin supports exactly one administrator. Authentication uses a fixed
email, bcrypt password hash, TOTP, one-time recovery codes, short-lived secure
sessions, login rate limiting, and audit events for login and every sensitive
mutation.

The management UI follows the existing industrial-minimal design system:
flat surfaces, thin borders, 2-6 px radii, Inter and JetBrains Mono, blue
accent, light/dark themes, and no gradients.

## Operational Behavior

- Published records cannot be deleted; they can be archived or disabled.
- A failed publication leaves the previous main-app version active.
- Publication ids and nonces make retries idempotent and replay resistant.
- Disabling a credential profile blocks provider calls for every user bound to
  that profile.
- Disabling code redemption does not revoke users who already redeemed it.
- Revoking redeemed users is a separate, confirmed and audited operation.
- Destructive cleanup of existing users and personal keys is an explicit
  deployment command after backup, never an automatic schema migration.
